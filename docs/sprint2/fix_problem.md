# Sprint 2 输出质量问题记录

## 文档目的

本文记录 Sprint 2 当前实际运行中发现的输出质量问题，用于后续需求分析和修复验收。

本文只描述：

- 当前出现的现象。
- 具有代表性的错误示例。
- 问题对用户体验和输出质量的影响。
- 期望达到的产品效果。

本文不包含技术原因、架构调整、代码实现或具体修复方案。

## 1. 输出中泄露内部 Evidence ID

### 当前现象

简历要点、面试回答和风险提示等面向用户的文本中，仍可能出现系统内部使用的 `evidence_ids`，例如：

```text
(evidence_ids: ["req_3:evidence:2", "req_4:evidence:2", "req_7:evidence:1"])
```

部分情况下还会出现无效的空 ID：

```text
(evidence_ids: ["", "", ""])
```

### 问题影响

- `evidence_ids` 是系统内部追踪信息，对普通用户没有可读价值。
- 内部 ID 会破坏简历要点、面试答案和风险提示的完整性与自然度。
- 空 ID 会进一步降低用户对系统结果可靠性的信任。
- 用户可能误以为这些 ID 需要保留在简历或面试回答中。

### 期望行为

- 所有面向用户展示的内容都不得出现 `evidence_ids`、requirement ID、chunk ID 等内部标识。
- 内部证据关联可以由系统保留，但不应拼接到简历要点、问题、答案、风险提示或建议文本中。
- 最终输出应当是可以直接阅读、修改或使用的自然语言内容。

## 2. JD 相关问题过于表面，不符合真实面试场景

### 当前现象

系统倾向于把 JD 中的资格要求直接改写成“你如何满足该要求”的问题，例如：

```text
你如何满足计算机背景硕士或者博士的要求？
```

```text
你如何满足岗位对具备扎实的编程基础（Python/C++/Java），熟悉常用算法与数据结构的要求？
```

这类问题只是复述 JD，没有形成真实面试中具有判断价值的问题。

### 问题影响

- 学历、专业背景和技能关键词通常可以由面试官直接从简历中确认，没有必要再次以低信息量问题询问。
- 问题无法检验候选人是否真正理解相关技术，也不能区分候选人的能力深度。
- 问题缺乏实际业务场景、系统设计背景、技术约束和决策空间。
- 用户无法通过这类问题进行有效的技术面试准备。

### 期望行为

- JD 相关问题应从岗位要求推导出具有专业性的技术问题，而不是简单复述资格条件。
- 对学历、专业背景等静态信息，如果简历中已经明确体现，不应生成“如何满足”式问题。
- 对 Python、C++、Java、算法与数据结构等基础要求，应进一步考察实际运用、设计选择、性能权衡、故障处理或工程实践。
- 问题应尽量带有明确的业务目标、技术场景和约束条件。
- 例如，当岗位涉及多模态平台时，应围绕平台要解决的问题、数据与模型选择、系统架构、评估方式、性能瓶颈和工程权衡提出具体问题。
- 最终问题应接近真实技术面试中的开放式专业问题，能够考察候选人的分析、设计和解决问题能力。

## 3. 简历深挖问题冗余且缺少针对性

### 3.1 问题中大段复制简历原文

#### 当前现象

系统生成简历深挖问题时，会在问题后直接拼接一大段项目或实习原文。原文可能包含完整的项目介绍、个人贡献、技术方案和项目成果，导致问题异常冗长。

典型形式如下：

```text
请深入介绍该项目中的个人职责、技术难点、结果和复盘：
（随后大段复制简历中的项目内容）
```

#### 问题影响

- 问题本身被简历原文淹没，用户难以快速识别面试官真正想问什么。
- 同一段简历内容可能被多次重复展示，造成严重冗余。
- 问题不像真实面试官的表达方式，更像系统把检索结果直接拼接到了问题后面。

#### 期望行为

- 问题只需要使用项目名称、公司名称、岗位名称或一句简短摘要来指明所讨论的经历。
- 不应在问题中完整复制项目介绍、个人贡献和项目成果。
- 如果简历中存在项目名称，应优先使用项目名称定位该项目。
- 如果没有明确名称，可以使用简洁、可辨认的经历摘要，但不能粘贴整段原文。

### 3.2 问题模板单一，缺乏专业深挖

#### 当前现象

不同项目和实习经历生成的问题高度相似，主要使用“请深入介绍……”这一固定模板，没有根据经历中的技术内容形成有针对性的追问。

### 问题影响

- 不同项目之间的技术差异没有得到体现。
- 问题无法覆盖候选人在项目中的真实职责、技术选择和工程判断。
- 用户无法针对项目中的薄弱环节或高价值亮点进行准备。

### 期望行为

- 项目问题应结合项目目标、个人职责、技术栈、技术难点、方案选择、实验或评估方法、结果和复盘进行提问。
- 实习问题应结合业务背景、具体工作、使用的技术、协作过程、交付成果和实际影响进行提问。
- 问题可以分别关注：
  - 使用了哪些技术栈，为什么选择这些技术。
  - 解决了什么具体问题。
  - 个人负责了哪些关键工作。
  - 遇到了哪些困难或失败，如何定位和解决。
  - 如何评估方案效果，指标是否合理。
  - 最终达到了什么成果，对项目或业务产生了什么影响。
  - 从项目中获得了哪些经验，如果重新实施会如何改进。
- 不同问题应有明确侧重点，避免对所有经历机械套用同一个问题模板。

## 4. JD 相关问题的答案缺乏概括、扩展和针对性

### 当前现象

JD 相关问题的示范答案经常直接复制一大段简历项目经历，缺少围绕问题进行重新组织和解释。

答案主要在重复“做过什么”，没有清晰说明这些经历为什么能够回答当前问题，也没有从已有经历扩展到岗位对应的技术场景。

### 问题影响

- 答案篇幅很长，但有效信息密度较低。
- 回答与问题之间的逻辑关系不清晰。
- 用户难以直接将答案用于真实面试表达。
- 答案没有体现候选人如何把已有经验迁移到目标岗位或新业务场景中。

### 期望行为

- 答案应首先直接回答问题，再选取最相关的一段或两段经历作为证据。
- 简历经历应经过概括和重新组织，而不是原文复制。
- 答案需要解释经历与 JD 要求之间的关联，并在此基础上进行合理扩展。
- 对场景型或系统设计型问题，答案应包含问题理解、解决思路、技术选择、权衡、验证方式和风险考虑。
- 回答应有清晰结构、自然表达和明确重点，既基于真实经历，又能体现候选人的专业思考能力。

## 5. 简历相关问题的答案机械复制原文

### 当前现象

简历深挖问题的答案同样会大段复制简历内容，项目介绍、个人贡献和结果几乎原封不动地再次出现。

答案没有根据具体问题重新组织内容，也缺少自然的口语化表达和重点取舍。

### 问题影响

- 答案与问题的对应关系较弱。
- 大段书面简历文本不符合真实面试中的口头表达习惯。
- 用户无法看出面对不同追问时应该选择哪些信息、如何组织回答。
- 相同经历在多个答案中反复出现，造成重复和阅读疲劳。

### 期望行为

- 答案应针对当前问题选择相关信息，不应完整复述整段项目或实习经历。
- 内容应重新组织为自然、连贯、适合面试表达的语言。
- 回答应突出候选人的个人贡献，避免只描述团队或项目整体情况。
- 根据问题类型，分别突出技术方案、困难与解决过程、结果、协作、复盘或经验迁移。
- 答案可以引用简历中的真实事实和指标，但必须围绕问题进行取舍和解释。

## 6. 风险提示判断过于机械，未正确理解简历语义

### 6.1 把基础技能要求机械地判定为缺少项目支撑

#### 当前现象

系统可能给出以下风险：

```text
关键岗位要求缺少项目或实习支撑
对应 JD 要求：具备扎实的编程基础（Python/C++/Java），熟悉常用算法与数据结构
简历现状：当前项目或实习经历中未找到直接支撑该要求的具体内容。
```

但是候选人的多个项目已经使用 Python、机器学习和深度学习工具完成了实际任务。这些经历本身已经能够间接或直接证明编程实践能力，不应要求候选人额外增加一个专门证明“会使用 Python”的项目。

#### 问题影响

- 风险判断忽略了项目技术栈和实际工作对基础能力的证明作用。
- 系统对所有 JD 要求使用过于僵硬的“必须有独立项目直接对应”标准。
- 给出的补充建议可能不符合真实简历写作习惯。

#### 期望行为

- 基础编程能力可以由多个项目、实习工作、技术栈和具体实现共同证明。
- 不应要求每一项基础技能都对应一个独立项目。
- 风险判断应综合技能列表、项目技术栈、个人贡献和实际任务，而不是只寻找完全相同的文字表述。

### 6.2 无法识别项目与 JD 领域要求之间的语义关联

#### 当前现象

系统可能把以下要求判定为未覆盖：

```text
掌握机器学习基础理论，了解搜、推、广系统、自然语言处理、多模态等至少一个领域。
```

但简历中已经存在语义分割、NLP 客户反馈分类、大语言模型与 RAG、腾讯混元多模态团队实习等高度相关经历。

#### 问题影响

- 系统依赖表面词语匹配，无法正确理解经历所属的技术领域。
- 明明存在强相关证据，仍被错误标记为“JD 未覆盖”。
- 风险提示与用户对自身经历的基本认知相冲突，严重影响结果可信度。

#### 期望行为

- 风险判断应理解项目、技术栈、任务类型和 JD 领域要求之间的语义关系。
- 语义分割应被识别为计算机视觉和机器学习相关实践。
- 文本分类、大语言模型和 RAG 应被识别为自然语言处理相关实践。
- 图像转文本评估和腾讯混元多模态团队实习应被识别为多模态相关经历。
- 当简历已经存在明确或高度相关的经历时，不应将对应 JD 要求判定为未覆盖。

### 6.3 风险原因出现无法理解或明显错误的数字判断

#### 当前现象

风险提示中可能出现：

```text
风险原因：数字 4 没有出现在引用的证据中。
```

该数字与用户看到的简历内容、风险标题和 JD 要求之间缺少清晰联系，用户无法理解数字来源，也无法判断应该如何修改简历。

### 问题影响

- 风险原因不可解释，缺乏上下文。
- 用户无法确认系统检测到了什么具体问题。
- 无法根据提示采取有效行动。

### 期望行为

- 数字风险必须明确指出对应的生成表述、证据内容以及两者之间的差异。
- 如果数字来自编号、列表序号、日期或其他非成果指标，不应被误判为未经证据支持的量化成果。
- 无法明确解释来源和影响的数字风险不应展示给用户。

### 6.4 风险提示中的简历现状仍包含内部 ID

#### 当前现象

风险提示可能显示：

```text
简历现状：在腾讯混元部门多模态团队实习期间，设计图像转文本模型的智能评估流程……
(evidence_ids: ["", "", ""])
```

### 问题影响

- 内部字段破坏风险提示的可读性。
- 空 ID 让结果显得不完整或存在系统错误。
- 用户容易误解为简历中需要保留的内容。

### 期望行为

- 风险提示的标题、JD 要求、简历现状、风险原因和建议中都不得出现内部 ID。
- 简历现状应当是对真实简历内容的简洁概括，不应包含系统元数据。

## 7. 统一验收标准

后续改进至少需要满足以下产品验收标准：

1. 简历要点、问题、答案、风险提示和建议中不出现任何内部 evidence、requirement 或 chunk ID。
2. JD 相关问题不简单复述学历、技能关键词或资格要求，而是形成具有业务场景和技术深度的专业问题。
3. 简历深挖问题只使用项目名、公司名、岗位名或简短摘要定位经历，不复制整段简历。
4. 不同项目和实习的问题具有不同侧重点，不机械使用同一个模板。
5. JD 相关答案先直接回答问题，再使用相关经历支撑，并能扩展到目标岗位场景。
6. 简历深挖答案根据具体问题重新组织语言，适合真实面试表达，不原封不动复制简历。
7. 风险判断能够综合技能、项目技术栈、个人贡献、实习任务和成果理解候选人能力。
8. 风险判断能够识别语义分割、NLP、大语言模型、RAG 和多模态经历与相关 JD 领域要求之间的联系。
9. 不要求每一项基础能力都必须拥有独立项目证明。
10. 数字风险必须有明确、可理解、可追溯的上下文，编号、日期等非成果数字不得被误判。
11. 相同内容不应在多个问题、答案或风险提示中大段重复。
12. 最终输出应当具体、自然、可解释，并且能够直接用于真实求职准备。
