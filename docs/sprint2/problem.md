# Sprint 2 问题记录与需求确认

## 背景

当前基础版 Career Agent 已能完成从简历材料和 JD 到分析结果的基本流程，但实际页面输出与预期存在偏差。本文只记录已发现问题和待确认需求，不包含代码改动方案。

## 1. 简历要点问题

### 已发现问题

- 网页展示中的简历要点只摘要了实习工作的部分内容。
- 简历要点没有摘要实习对应的公司名和岗位信息。
- 简历要点没有摘要平时项目内容。
- 当前输出反而摘要了一些简历中列出的技能项。
- 相比技能列表，项目经历和实习经历对求职材料更重要，当前优先级不符合预期。

### 待确认问题

1. 简历要点是否应该优先从「实习经历」和「项目经历」中生成，而不是从技能列表中生成？
   对，简历要点优先从实习经历和项目经历中生成。
2. 实习经历生成要点时，是否必须包含公司名、岗位名、职责/项目内容、成果或技术栈？
   需要包含公司名、项目内容、成果和技术栈
3. 项目经历生成要点时，是否必须包含项目名称、项目目标、个人贡献、技术栈、结果或可量化影响？
   对，需要包含
4. 技能列表是否只作为辅助信息，用于补充项目/实习表述，而不单独生成简历要点？
   对，技能里表作为辅助信息
5. 如果简历中项目和实习很多，简历要点应该生成几条？是否需要按 JD 匹配度排序？
   按照和 JD 匹配度排序生成个三条

## 2. 求职信草稿问题

### 已发现问题

- 当前求职信草稿只生成了几句话。
- 当前输出与预期的 cover letter 初稿不符合。
- 当前求职信草稿价值较低，用户倾向于删除该模块。

### 待确认问题

1. 是否在 Sprint 2 中直接删除「求职信草稿」模块？
   删掉
2. 删除后，是否需要从后端结果、前端展示、测试和文案中都移除 cover letter？
   是
3. 是否需要保留后端字段但前端不展示，还是彻底从产品功能中移除？
   彻底删除
4. 删除求职信后，页面是否需要把空间让给「简历要点」「面试准备」「风险提示」等更重要模块？
   对，把空间让给其他部分

## 3. 面试准备问题

### 已发现问题

- 当前面试准备问题不够贴合 JD。
- 当前问题没有清晰区分面试官基于 JD 要求可能会问的问题。
- 当前问题没有充分覆盖面试官基于简历项目或实习经历可能会问的问题。
- 当前准备建议过于模板化。
- 当前准备建议经常类似于：“这个话题可以把个人经历和目标岗位要求直接连接起来。准备一段简洁的项目 walkthrough，并明确说明能力点对应的材料证据。”
- 这种建议没有结合具体简历内容、JD 要求和对应问题生成可直接参考的回答。

### 期望方向

- 面试准备应包含与 JD 要求相关的问题。
- 面试准备应包含根据简历项目或实习经历推导出的可能问题。
- 准备建议应改为结合「简历内容 + JD 要求 + 面试问题」生成的回答思路或示范回答。

### 待确认问题

1. 面试准备是否需要分成两类展示：`JD 相关问题` 和 `简历深挖问题`？
   分成两类展示
2. 每类问题各生成几条比较合适？
   每个问题：对于简历深挖问题，根据简历里项目经历和实习经历来，如果内容很多，那么生成三四条，不然就一两条。JD 相关问题也是根据 JD 要求来，如果要求比较多就三四条，不然就一两条。
3. 回答内容应是完整示范回答，还是回答框架/要点？
   完整的示范回答
4. 回答是否需要引用证据 ID，方便追溯到简历项目或实习经历？
   ID 不需要，只需要引用相关经历即可
5. 对项目经历问题，是否需要覆盖项目背景、个人职责、技术难点、结果、复盘？
   重点侧重于个人职责，技术难点，结果和复盘
6. 对实习经历问题，是否需要覆盖公司/岗位、业务场景、职责、协作、成果？
   只需要覆盖技术，工作内容和成果
7. 是否需要根据 JD 的高优先级要求优先生成面试问题？
   需要

## 4. 风险提示问题

### 已发现问题

- 风险提示里显示“未覆盖要求：req_1”，这个 ID 对用户不友好，也无法体现具体 JD 内容。
- 每条风险提示内容高度重复。
- 当前常见提示是：“高优先级岗位要求没有被生成的简历要点覆盖。”
- 该提示不够具体，不能说明 JD 中具体哪一部分没有被简历覆盖。
- 当前风险评估没有充分结合 JD 内容和简历内容。
- 当前风险评估没有指出简历中哪些内容表述太泛。
- 当前风险提示数量可能过多或过宽泛。

### 期望方向

- 风险评估应基于 JD 内容和简历内容，指出具体缺口。
- 风险评估应说明 JD 中哪一部分在简历里没有体现。
- 风险评估也应说明简历中哪些内容表述太泛，需要补充细节。
- 风险提示只保留最重要的 3 条。
- 风险提示应避免展示 `req_1` 这类内部 ID，改为展示用户可理解的 JD 要求文本。

### 待确认问题

1. 风险提示是否固定最多展示 3 条？
   对
2. 3 条风险提示是否按严重程度和求职影响排序？
   是的
3. 每条风险提示是否需要包含：风险标题、对应 JD 要求、简历现状、为什么有风险、建议如何补充？
   需要
4. 是否需要区分风险类型，例如 `JD 未覆盖`、`简历表述太泛`、`证据不足`、`生成内容可能夸大`？
   需要
5. 是否完全隐藏内部 requirement ID，只展示 JD 原文或简短摘要？
   是的
6. 对“简历表述太泛”的判断，是否应优先针对项目经历和实习经历，而不是技能列表？
   是的，技能列表这类表示自己掌握了什么技能的部分基本不重要
